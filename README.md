# MLOps Pipeline

**Проект:** ML-системы обработки потоковых данных.  
**Набор данных:** Rain in Australia.

**[Подробная документация проекта](./doc)**

---

## 1. Архитектура ML-конвейера (Задание 1)
Архитектура включает:
1. `data_collection` - сбор потоковых батчей (по 25 000 строк) и расчет метапараметров.
2. `data_quality` - проверка данных через 5 логических правил и очистка пропусков.
3. `model_training` - сборка sklearn Pipeline (SimpleImputer + Encoder/Scaler) и обучение Дерева Решений и Нейросети (MLPClassifier).
4. `model_validation` - применение TimeSeriesSplit для валидации моделей с учетом времени.
5. `model_inference` - применение лучшей модели к новым данным.
6. `report_generator` - формирование отчета.

## 2. CI/CD Workflow (Задание 2)

### Инструкции по развертыванию (Deployment Instructions)
Проект поддерживает автоматическое развертывание и тестирование через **GitHub Actions**.

**Для локального запуска:**
1. Склонируйте репозиторий: `git clone https://github.com/n0thingSp3zial/msu_mlops.git`
2. Установите зависимости: `pip install -r requirements.txt`
3. Запустите пайплайн: `python run.py -mode update`

**Для работы через облако (GitHub Actions):**
Развертывание происходит автоматически. При любом коммите в ветку `main` (push/pull request) GitHub Actions поднимает виртуальную машину `ubuntu-latest`, устанавливает зависимости и прогоняет конвейер. Обученная модель и логи становятся доступны для скачивания во вкладке **Actions -> Artifacts**.

### Аргументация выбранного подхода
Для реализации CI/CD был выбран **"Базовый сценарий"** с рядом архитектурных расширений:
* **Триггер:** Workflow запускается по событию `on: push`. Это гарантирует, что любое изменение в коде немедленно тестируется на актуальных данных.
* **Тестирование:** Перед запуском обучения добавлен шаг проверки целостности ML-пайплайна с помощью библиотеки `pytest`. Это предотвращает запуск длительного обучения, если структура конвейера была нарушена разработчиком.
* **Логирование:** Вывод системы переведен на профессиональную библиотеку `logging`. Текстовый файл лога `pipeline.log` выгружается как артефакт для мониторинга (Обязательное требование).
* **Сериализация:** Итоговая модель (`best_model.pkl`) выгружается из GitHub Actions в виде артефакта. Это позволяет внешним микросервисам или пользователям скачивать актуальную модель без необходимости обучать её локально.
